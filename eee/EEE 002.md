# **EEE 002: The Provable Core**

**Author:** Vitali

**Status:** Draft

**Created:** 2026-03-15

**Depends on:** EEE 001

## **Abstract**

This EEE defines the first implementation milestone for Eminix: a minimal but demonstrable subset of the architecture specified in EEE 001. The goal is to prove the core event-driven actor model works, that the performance SLOs are achievable, and that the foundation is sound enough to build on. This is not a usable editor -- it is a **provable systems foundation** with benchmarks.

## **Motivation**

EEE 001 specifies a large architecture. Attempting to build it all at once risks never shipping anything. This milestone isolates the smallest subset that:

1. Exercises the critical architectural patterns (event bus, actors, priority QoS).
2. Demonstrates measurable performance characteristics.
3. Produces a working artifact: accepts keystrokes, updates a buffer, renders output.

## **Scope**

### In Scope

1. **Event system with QoS** -- Critical (lossless, bounded, backpressure) and Ephemeral (lossy, broadcast) channels working together.
2. **Actor runtime** -- A minimal actor abstraction. Actors receive events via priority-aware mailboxes. At least three actors: InputReader, BufferActor, Renderer.
3. **Gap buffer** -- A correct, tested gap buffer implementation with insert, delete, and cursor movement.
4. **End-to-end keystroke pipeline** -- Terminal input → InputReader actor → Critical event → BufferActor → buffer mutation → Ephemeral render event → Renderer actor → terminal output. One complete loop through the architecture.
5. **Benchmarks** -- Latency benchmarks proving:
   - Gap buffer insert/delete operations.
   - Event bus publish/subscribe round-trip for ephemeral events.
   - Critical channel send/receive latency.
   - End-to-end keystroke-to-buffer-update latency (target: p99 < 1ms).

### Out of Scope

- Process isolation (UI and core run in the same process).
- Scripting engine.
- WAL, crash recovery, atomic saves.
- File I/O (buffer is memory-only).
- Keybinding layers and command system.
- TUI framework (raw terminal output is sufficient).
- Undo/redo.

## **Technical Specification**

### **1. Event System**

Two channel types, matching the QoS model from EEE 001:

- **Ephemeral channel:** lossy under load, multi-consumer broadcast. Used for render notifications and cursor updates.
- **Critical channel:** bounded with backpressure, one per actor. This is the actor's "mailbox" for events that must not be dropped.

The event model must support at minimum these event kinds:

| Event | Priority |
|---|---|
| KeyPress | Critical |
| Delete | Critical |
| CursorMove | Critical |
| Shutdown | Critical |
| BufferUpdated | Ephemeral |
| CursorMoved | Ephemeral |

### **2. Actor Runtime**

Each actor is an isolated async task with its own mailbox. An actor owns:
- A bounded channel receiver for critical events (its "mailbox").
- An optional subscription to the ephemeral event bus.

The actor's event loop must always prioritize critical events over ephemeral ones.

### **3. Actors**

| Actor | Receives (Critical) | Emits |
|---|---|---|
| **InputReader** | Terminal raw input | `KeyPress`, `Delete`, `CursorMove`, `Shutdown` (all Critical, sent to BufferActor's mailbox) |
| **BufferActor** | `KeyPress`, `Delete`, `CursorMove` | `BufferUpdated`, `CursorMoved` (Ephemeral, via EventBus) |
| **Renderer** | None | None (subscribes to Ephemeral EventBus, writes to terminal) |

### **4. Gap Buffer**

A gap buffer supporting insert, delete, and cursor movement. No rope, no piece table -- a gap buffer is simple enough to implement correctly and fast enough for this milestone. The API must support at minimum: inserting a character at the cursor, deleting the character before the cursor, moving the cursor left/right, and retrieving the full buffer contents.

### **5. Benchmarks**

| Benchmark | Target |
|---|---|
| Gap buffer: 10k sequential inserts | Baseline (establish, no SLO yet) |
| Gap buffer: insert at random positions | Baseline |
| Ephemeral event bus: publish + receive round-trip | p99 < 100us |
| Critical channel: send + receive | p99 < 50us |
| End-to-end: simulated keypress → buffer update | p99 < 1ms |

### **6. Tests**

- Gap buffer: correctness of insert, delete, cursor movement, edge cases (empty buffer, cursor at boundaries).
- Event system: Critical events delivered in order, Ephemeral events droppable under load, panic on Critical-via-broadcast.
- Actor integration: keypress sequence produces expected buffer content.

## **Definition of Done**

1. `cargo test` passes with all tests green.
2. `cargo clippy` produces no warnings.
3. `cargo bench` runs and produces latency histograms.
4. Running `cargo run` drops into raw terminal mode, accepts keystrokes, displays them, and exits cleanly on Ctrl-C.
5. README.md documents architecture, how to run, and benchmark results.

## **What This Proves**

- The event-driven actor model works in practice, not just on paper.
- The QoS split (Critical/Ephemeral) is implementable and measurable.
- Async Rust can meet the sub-millisecond latency targets.
- The author can write idiomatic, tested, benchmarked Rust systems code.
